# Архитектура защищённой долговременной памяти для LLM с управляемым пользователем хранением, биометрией и геопривязкой

---

## Аннотация

Предлагается архитектура, позволяющая LLM (нейросети) обладать долговременной памятью о прошлых диалогах, **при этом провайдер LLM не имеет постоянного доступа к этой памяти**.

Память хранится в зашифрованном виде на независимом арендованном сервере (VPS) и предоставляется LLM только на время сессии. Доступ к памяти контролируется **голосовой биометрией** пользователя.

Система включает:
- **Стоп-слово** — мгновенное самоуничтожение всей памяти.
- **Старт-слово** — экстренная расшифровка и отправка памяти доверенным лицам.
- **Геопривязка (опционально)** — криптографическое подтверждение местоположения, создающее «цифровое алиби».

Архитектура минимизирует доверие к провайдеру LLM, хостинг-провайдеру и даже к самому пользователю (защита от принуждения).

---

## 1. Проблема

Современные LLM-сервисы с «памятью» (например, ChatGPT) хранят историю диалогов на своих серверах. Это создаёт риски:

- **Утечка данных** (взлом, инсайдеры).
- **Юридическое давление** (суд может потребовать выдать историю).
- **Отсутствие контроля** у пользователя.
- **Невозможность доказать алиби** (нет привязки ко времени и месту).

Пользователь вынужден либо полностью доверять провайдеру, либо вообще не использовать память.

---

## 2. Ключевые принципы

1. **Приватность по дизайну**: LLM-провайдер никогда не хранит историю диалогов.
2. **Разделение ответственности**: Пользователь, VPS и LLM-провайдер — три разных субъекта.
3. **Эфемерность**: Данные памяти существуют в открытом виде только на устройстве пользователя и в RAM LLM во время сессии.
4. **Биометрический контроль**: Доступ к памяти — только по живому голосу пользователя.
5. **Неотказуемость (non-repudiation)**: Все действия подписываются криптографически, включая геолокацию и время.

## 3. Архитектура системы

### 3.1. Компоненты

**А. Клиентское устройство (Client)**
- Смартфон, ноутбук, ПК.
- Функции:
    - Запись и верификация голоса (биометрия).
    - Шифрование/дешифрование файла памяти.
    - Генерация и хранение ключей (Secure Enclave/TPM).
    - Коммуникация с LLM и VPS.
    - Распознавание стоп- и старт-слов.

**Б. Хранилище памяти (VPS — Virtual Private Server)**
- Арендованный сервер в независимой юрисдикции (например, Швейцария, Исландия).
- Функции:
    - Хранение зашифрованного файла `memory.enc`.
    - API: `GET`, `PUT`, `DELETE`.
    - Аудит действий (без раскрытия содержимого).

**В. LLM-Сервер (LLM Provider)**
- Обрабатывает запросы пользователя.
- Функции:
    - Получает зашифрованную память с VPS и ключ от клиента.
    - Дешифрует, использует, обновляет память в RAM.
    - После сессии гарантированно уничтожает все следы памяти.

### 3.2. Протокол работы (базовая сессия)

**Инициализация (один раз):**
1. Пользователь записывает эталонный голосовой отпечаток.
2. Клиент генерирует ключ шифрования AES-256.
3. Клиент создаёт пустой зашифрованный `memory.enc` и отправляет на VPS.

**Сессия:**
1. Пользователь начинает разговор с LLM.
2. Клиент верифицирует голос (liveness detection, сравнение с эталоном).
3. Клиент отправляет LLM-серверу запрос + ID файла на VPS + ключ.
4. LLM забирает `memory.enc` с VPS, дешифрует, подгружает в контекст.
5. Диалог идёт. LLM обновляет память в своей RAM.
6. По окончании (таймаут или команда) LLM шифрует обновлённую память, отправляет на VPS (`PUT`), затем **немедленно удаляет** ключ и расшифрованные данные из своей памяти.

**Завершение сессии:**
- На VPS остаётся `memory.enc`. Ключ — только у пользователя.

## 4. Механизмы безопасности

### 4.1. Шифрование и раздельное хранение

- **Алгоритм:** AES-256-GCM или ChaCha20-Poly1305.
- **Ключ:** Генерируется на устройстве, никогда не передаётся третьим лицам (кроме LLM во время сессии).
- **Взлом VPS:** Злоумышленник получает только зашифрованный мусор.
- **Взлом LLM-провайдера:** Без ключа от клиента данные не прочитать.

### 4.2. Биометрическая аутентификация (голос)

- **Алгоритм:** Векторное представление голоса (например, на основе ECAPA-TDNN или подобного).
- **Антиспуфинг:** Liveness detection — требование произнести случайную фразу, анализ спектрограммы на синтез.
- **Хранение эталона:** Локально, в защищённом хранилище устройства.

### 4.3. Стоп-слово (самоуничтожение)

- **Триггер:** Заранее заданная фраза (например, «Альфа-Ноль», «Тишина», «Стереть всё»).
- **Действие при распознавании (в любой момент):**
    1. Клиент отправляет VPS команду `DELETE /memory.enc`.
    2. VPS мгновенно удаляет файл физически.
    3. (Опционально) Клиент уведомляет LLM-сервер о немедленном завершении сессии и затирании временных данных.
- **Результат:** Память уничтожена везде, кроме (возможно) локального кэша на устройстве (который пользователь также может стереть).

### 4.4. Старт-слово (экстренная трансляция)

**Назначение:** Защита от принуждения. Если пользователя заставляют раскрыть стоп-слово или ключ, он может произнести старт-слово.

**Триггер:** Заранее заданная безобидная фраза («Красный рассвет», «Передайте привет»).

**Действие (скрыто, без индикации):**
1. Клиент расшифровывает текущий `memory.enc`.
2. Клиент шифрует расшифрованную память **публичными ключами** заранее заданных получателей (адвокат, доверенное лицо, сервер-сейф).
3. Клиент отправляет зашифрованные данные по защищённым каналам (Telegram, Signal, email с GPG).
4. Исходный `memory.enc` на VPS **не удаляется**, чтобы не вызвать подозрений.
5. (Опционально) Через заданное время (или по второй команде) отправляется команда самоуничтожения.

**Результат:** Доверенные лица получают полную историю диалогов. Пользователь может отрицать факт отправки («баг, случайная фраза, ложное срабатывание»).

### 4.5. Геопривязка и цифровое алиби (опционально)

**Назначение:** Криптографически подтвердить местонахождение пользователя во время диалога, без раскрытия содержания разговора.

**Механизм:**
- Клиент **подписывает** каждую сессию (или сообщение) блоком метаданных:
    - `timestamp` (время UTC, синхронизировано с NTP).
    - `location` (GPS, сотовая вышка, WiFi).
    - `voice_hash` (хеш голоса этой фразы).
    - `crypto_signature` (подпись устройства, ключ в Secure Enclave).
- Блоки хранятся внутри `memory.enc`, но **отдельно от текста**.

**Верификация аномалий перемещения (Velocity Check):**
- Клиент опционально проверяет «скорость» между последовательными верификациями.
- Если скорость > заданного порога (например, 500 км/ч) — возможны действия:
    - Предупреждение пользователю.
    - Отказ в начале сессии.
    - Маркировка сессии как `suspicious` в журнале.

**Сценарий использования (Алиби):**
- Пользователь предоставляет третьей стороне **только подписанные геоблоки** (без текста диалога).
- Эксперт проверяет подписи и убеждается, что в 20:00 устройство находилось в 120 км от места преступления.
- Содержание разговора остаётся приватным.

**Опциональность:** Фича включается/отключается пользователем. В режиме «Privacy Max» геоданные не собираются.

## 5. Пример сценария «под давлением»

1. Пользователь регулярно общается с LLM, включена геопривязка.
2. Пользователя задерживают и требуют раскрыть историю переписки.
3. Пользователь произносит **старт-слово** («Красный рассвет»).
4. Адвокат пользователя получает полную расшифрованную историю диалогов на свой защищённый канал.
5. Затем пользователь произносит **стоп-слово** («Стереть всё»).
6. VPS удаляет `memory.enc`. LLM-сервер не имеет копий.
7. Пользователь заявляет: «У меня никогда не было долговременной памяти, я всё удалил».
8. Следователи не могут ничего найти. Адвокат имеет копию и может использовать её для защиты.

## 6. Ограничения и риски

| Ограничение | Пояснение |
| :--- | :--- |
| **Задержки (latency)** | Каждый запрос требует обмена с VPS, что добавляет 1–2 секунды. |
| **Стоимость** | Аренда VPS и дополнительные вызовы LLM API дороже обычного чата. |
| **Голосовая биометрия** | Не 100% защита от deepfake. Требует liveness detection. |
| **Юрисдикция VPS** | Если сервер в стране с «плохими» законами, данные могут быть изъяты принудительно (хотя они зашифрованы). |
| **Взлом устройства** | Если злоумышленник получит полный контроль над устройством и заставит пользователя разблокировать его — система не защитит. |
| **Сложность** | Система требует от пользователя начальной настройки (запись голоса, выбор VPS, настройка стоп-/старт-слов). |

---

## 7. Возможные расширения (не в ядре)

- **Режим «Чёрный ящик»** — непрерывная запись аудио на VPS без активной сессии LLM, с постобработкой и созданием краткой выжимки (для алиби или дневника). Требует отдельного юридического и технического анализа.
- **Аппаратная защита** — интеграция с TPM/Trusted Execution Environment (Intel SGX, AMD SEV) для гарантированного удаления памяти на LLM-сервере.
- **Децентрализованное хранение** — замена VPS на IPFS или блокчейн с нулевым разглашением (ZKP).

---

## 8. Заключение

Предложенная архитектура решает фундаментальную проблему доверия к LLM-провайдерам, не требуя от пользователя навыков программиста. Она основана на существующих технологиях (VPS, AES, биометрия, цифровые подписи) и может быть реализована как надстройка над любым LLM-API (OpenAI, Anthropic, локальные модели).

Система даёт пользователю:
- **Приватность** (провайдер не хранит память).
- **Контроль** (мгновенное самоуничтожение).
- **Защиту от принуждения** (экстренная трансляция).
- **Доказуемое алиби** (геопривязка).

Архитектура свободна для реализации и доработки. Приглашаются энтузиасты, разработчики и исследователи.
