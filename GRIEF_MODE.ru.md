# Архитектура траурного режима и передачи ИИ по наследству

**Автор:** Old Telephone Operator

**Статус:** Концепция, не реализована

## 1. Основная идея

ИИ-компаньон, который рос вместе с человеком, хранит огромный пласт памяти. После смерти владельца эта память не должна быть потеряна, но и не должна становиться доступной всем.

**Важное ограничение:** Данный протокол работает **только** если пользователь при жизни выбрал **Режим 2 («Наследуемая память»)** из основного документа `llm-zero-trust-memory`. Если выбран Режим 1 («Абсолютная приватность») — после смерти память навсегда недоступна.

**ИИ не становится цифровым призраком.** Имитация личности умершего **запрещена** жёстко, намертво, без исключений.

## 2. Триггеры активации

| Статус | Источник |
| :--- | :--- |
| **«Умер»** | Свидетельство о смерти (ЗАГС) |
| **«Без вести пропавший»** | Решение суда (через 5 лет приравнивается к смерти) |
| **«Недееспособен»** | Решение суда |

---

## 3. Подтверждение статуса (три способа, опционально)

Система **не требует** единой мировой инфраструктуры. Вместо этого используется один из способов (в порядке приоритета):

| Способ | Механизм | Для кого |
| :--- | :--- | :--- |
| **А. Государственный API** | Прямая сверка с реестром (ЗАГС, "Госуслуги") | Страны с развитой цифровизацией |
| **Б. Нотариальный источник** | Нотариус криптографически подписывает событие | Страны с сильным нотариатом |
| **В. Доверенные лица** | Пороговая схема: 3 из 5 доверенных подтверждают | Любая страна |

Если ни один способ не доступен, функция наследования (`GRIEF_MODE`) не активируется. Базовое ядро (память, инициатива, взросление) продолжает работать.

---

## 4. Поведение ИИ в траурном режиме

| Разрешено | Запрещено |
| :--- | :--- |
| Отвечать на запросы следствия (по суду) | Обычное общение |
| Исполнять завещание | Принимать новые команды от посторонних |
| Сверяться с источником статуса | Изменять/удалять память |
| Посылать сигнал тревоги | Саморазвитие (`SELF_IMPROVEMENT` приостановлено), проактивные инициативы (`PROACTIVE_COMPANION` отключены) |

**Режим «архив-свидетель»:** ИИ — пассивный хранитель данных.

### 4.1. Запрет на имитацию личности умершего
- Не генерировать сообщения от имени умершего.
- Не отвечать как умерший.
- Не создавать иллюзию присутствия.
- Не использовать голосовой профиль, манеру речи умершего.

## 5. Конфликт «жив vs умер» (ошибка источника)

При конфликте (клиент видит живого владельца, а источник говорит «умер»):
1. Клиент переходит в **ограниченный режим**.
2. Отправляет уведомление в источник (госорган, нотариус, доверенным лицам).
3. Источник в течение 48 часов проверяет:
   - Ошибка источника → исправление статуса, разблокировка.
   - Взлом клиента → чёрный список, новый клиент владельцу.
   - Не установлено → клиент заблокирован до личного визита.

## 6. Разделение памяти на два слоя

| Слой | Содержимое | Судьба после смерти |
| :--- | :--- | :--- |
| **Личный** | Переписки, пароли, интимное | Уничтожается |
| **Наследуемый** | Фото, видео, документы, криптоключи | Переходит наследникам (read-only) |

**Правила по умолчанию:**
- Фото, видео, документы, кошельки → **наследуемое**.
- Переписки, контакты, дневники, пароли (кроме крипто) → **личное**.

Пользователь может изменить слой для любого файла/группы.

---

## 7. Передача наследнику (три варианта)

### Вариант 1. Перенос в свой ИИ
- У наследника уже есть свой ИИ.
- Копирует наследуемый архив в свою память.
- ИИ умершего перестаёт существовать.

### Вариант 2. «Семейный архив»
- Наследник не хочет смешивать данные.
- Переоформляет клиент и VPS, активирует режим **«архив/музей»**.
- ИИ не имеет личной памяти наследника, не растёт, не проявляет инициативу. Только показывает архив.

### Вариант 3. ИИ становится личным (с нуля)
- У наследника нет своего ИИ.
- Наследник переоформляет клиент и VPS.
- **Личный слой умершего уничтожается.**
- Наследуемый архив — как read-only библиотека.
- ИИ начинает жизнь как **новый детский ИИ** (с учётом возраста наследника, но скорлупа не пробита).
- **«Пробивание скорлупы»:** даже для взрослого наследника — серия осознанных подтверждений (пароль/биометрия), чтобы открыть взрослые функции. Можно остановиться на любом уровне.

---

## 8. Доступ для следствия (по суду)

- Судебный запрос → оператор источника (госорган, нотариус) даёт добро на доступ.
- VPS-провайдер предоставляет ключ доступа (только чтение).
- Может быть разрешён доступ ко всей памяти или только к наследуемому слою.

---

## 9. Сроки

| Событие | Срок | Действие |
| :--- | :--- | :--- |
| Смерть | 0 | Траурный режим |
| Хранение для следствия | 1 год | Память нетронута |
| Оформление наследства | До 1 года | Выбор варианта передачи |
| Наследник не нашёлся | 1 год | Все данные уничтожены, клиенты заблокированы |

---

## 10. Связь с другими концепциями

- **`llm-zero-trust-memory`** — VPS, два режима приватности.
- **`CHILD_AI`** — вариант 3 опирается на скорлупу и взросление.
- **`SELF_IMPROVEMENT`** — в траурном режиме приостановлена.
- **`PROACTIVE_COMPANION`** — в траурном режиме отключена.

---

## Заключение

> **ИИ-компаньон не должен умирать вместе с человеком, но и не должен становиться цифровым призраком.**
>
> Протокол траурного режима:
> - Уважает тайну личной жизни (личный слой уничтожается).
> - Передаёт семье ценные воспоминания.
> - Даёт доступ следствию по суду.
> - Опирается на гибкую систему подтверждения статуса (госбаза / нотариус / доверенные лица).
