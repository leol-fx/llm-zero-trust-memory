## 1. Основная идея

ИИ-компаньон, который рос вместе с человеком, хранит огромный пласт личной и семейной памяти. После смерти или недееспособности владельца эта память не должна быть потеряна безвозвратно, но и не должна становиться доступной всем подряд.

Предлагается **протокол траурного режима**, который:
- блокирует активное общение с ИИ
- переводит его в режим «архив-свидетель»
- сохраняет данные для следствия (по решению суда)
- позволяет передать наследуемую память (фото, документы, интеллектуальную собственность) новому владельцу
- **уничтожает личный слой** (переписки, пароли, интимные тайны) без возможности восстановления

Архитектура опирается на **государственную базу статусов владельцев** и **раздельное шифрование** личного и наследуемого слоёв памяти.

---

## 2. Триггеры активации траурного режима

ИИ переходит в траурный режим при получении из **государственной базы** одного из следующих статусов:

| Статус | Источник данных | Примечание |
| :--- | :--- | :--- |
| **«Умер»** | Свидетельство о смерти (ЗАГС) | Основной сценарий |
| **«Без вести пропавший»** | Решение суда | По истечении срока (например, 5 лет) приравнивается к смерти |
| **«Недееспособен»** | Решение суда | Деменция, кома, психическое расстройство |

**Важно:** Статус «пропал» требует особого внимания: если владелец объявится, ИИ должен быть разблокирован (см. раздел 5).

---

## 3. Государственная база статусов владельцев

- **Оператор:** Государственный орган (ЗАГС, Минцифры) или уполномоченная компания (нотариальная палата).
- **Данные:** ID владельца (привязан к ИИ, VPS, клиенту), статус, дата изменения, данные наследников (при наличии).
- **Обновление:** На основании официальных документов.
- **Защита:** Криптографическая подпись всех записей, аудит доступа.

## 4. Поведение ИИ в траурном режиме

| Что разрешено | Что запрещено |
| :--- | :--- |
| Отвечать на запросы полиции/спецслужб (по санкции суда) | Обычное общение с кем-либо |
| Исполнять завещание (разослать письма, файлы) | Принимать новые команды от посторонних |
| Сверяться с госбазой (статус, переоформление) | Изменять или удалять память (кроме случаев, оговоренных в завещании) |
| Посылать сигнал тревоги при попытке взлома | Саморазвитие, инициатива (IDLE_INITIATIVE отключается) |

**Режим «архив-свидетель»:**  
ИИ не инициирует диалог, не задаёт вопросов, не предлагает действий. Он — пассивный хранитель данных.

### 4.1. Запрет на имитацию личности умершего

**Железное правило:**  
ИИ после смерти владельца **не может и не должен** выполнять функции его двойника, имитации, цифрового призрака или заместителя.

**Что именно запрещено:**
- Генерировать сообщения от имени умершего (текст, голос, видео).
- Отвечать на запросы так, как если бы это делал умерший.
- Создавать иллюзию, что умерший продолжает общаться через ИИ.
- Использовать личные данные умершего (голосовой профиль, манеру речи, «личность») для активного взаимодействия с живыми людьми.

**Разрешено только:**
- Действовать как **семейный архив** (показывать фото, видео, документы, цитировать факты из жизни — но как архив, а не как личность).
- Выполнять формальные действия, явно указанные в завещании (разослать письма, перевести средства, передать файлы).
- После переоформления стать **новым ИИ** нового владельца (с чистым листом, без имитации предыдущего).

**Обоснование:** Этическая защита живых, юридическая чистота, техническая честность. Нарушение производителем этого запрета должно преследоваться по закону.

## 5. Механизм проверки и разрешения конфликта «жив vs умер»

При каждом включении (или периодически) клиент:
1. Проверяет наличие живого владельца (биометрия / пароль / ключ).
2. Запрашивает статус в госбазе.

**Конфликт:** локальная проверка говорит «владелец жив», а база отвечает «умер».

### Действия клиента:
- Немедленный переход в **ограниченный режим** (блокировка общения, кроме SOS и связи с госорганом).
- Отправка уведомления о конфликте в госорган (с ID владельца, временем, результатами проверки).
- Отображение сообщения владельцу: *«Обнаружен конфликт в базе. Обратитесь в [орган] для решения проблемы. Код конфликта: XXXX»*.

### Решение госоргана (в срок, например, 48 часов):
- **Ошибка базы** → статус исправляется на «жив», клиент получает команду разблокировки.
- **Взлом клиента** → статус остаётся «жив», клиент вносится в чёрный список, владелец получает новый.
- **Не установлено** → клиент остаётся заблокированным до личного визита владельца.

## 6. Разделение памяти на два слоя (ключевое требование)

| Слой | Содержимое | Ключ шифрования | Судьба после смерти |
| :--- | :--- | :--- | :--- |
| **Личный** | Переписки, пароли, интимные данные, «личное» по выбору владельца | Только у владельца | Безвозвратно уничтожается (через год или при переоформлении) |
| **Наследуемый** | Фото, видео, документы, интеллектуальная собственность | Может быть передан наследникам (через госбазу) | Переходит наследникам (режим read-only или импорт) |

Владелец **при жизни** определяет, какая информация относится к какому слою.

### 6.1. Интерфейс управления слоями (правила по умолчанию и исключения)

Чтобы пользователю не приходилось вручную классифицировать каждый файл, система использует **набор умолчаний**, основанных на типе данных. При этом пользователь (и только он) может **явно изменить слой** для любого файла или группы файлов.

#### Правила распределения по умолчанию

| Тип данных | Слой по умолчанию | Обоснование |
| :--- | :--- | :--- |
| **Фото, видео, аудиозаписи** | Наследуемый | Семейные архивы, память о человеке |
| **Документы (PDF, DOC, таблицы)** | Наследуемый | Интеллектуальная собственность, важные записи |
| **Информация о кошельках и криптоключах** | Наследуемый | Финансовое наследство |
| **Переписки (чат, голосовые сообщения)** | Личное (уничтожаемое) | Приватность диалогов |
| **Записная книжка, контакты, календарь** | Личное (уничтожаемое) | Личная и деловая жизнь, номера телефонов живых людей |
| **Дневниковые записи, заметки «для себя»** | Личное (уничтожаемое) | Интимная сфера |
| **Пароли, ключи доступа (кроме криптокошельков)** | Личное (уничтожаемое) | Высокий риск утечки |

#### Исключения

Пользователь может **явно изменить слой** для:
- **Конкретного файла** (через свойства: «Отнести к наследуемому / личному»)
- **Группы файлов** (выделить несколько, применить правило)
- **Целого типа** (например, «все фото из папки "Рабочие" — личные»)
- **По маске имени или дате** (например, «все фото до 2020 года — наследуемые, после — личные»)

**Важно:**  
- Изменение слоя применяется **только к будущему**, но может быть применено и к уже существующим файлам по желанию пользователя.
- Пользователь всегда может **передумать** и перенести файл из одного слоя в другой.
- После смерти владельца **никакие изменения слоёв невозможны** (только чтение для наследников).

#### Отображение в интерфейсе

- Два «хранилища» (папки): **«Наследуемое»** и **«Личное»**.
- При создании файла система предлагает выбор, но по умолчанию выбирает слой согласно таблице.
- Цветовая маркировка: зелёный — наследуемое, серый/красный — личное.
- Периодическое напоминание (раз в месяц): «Проверьте, правильно ли распределены ваши данные. Личное будет уничтожено, наследуемое перейдёт к наследникам».

## 7. Передача ИИ наследнику: три варианта

После оформления наследства (обычно в течение года) наследник может выбрать один из трёх сценариев.

### Вариант 1. Перенести архив в свой личный ИИ

- **Условие:** У наследника уже есть свой ИИ-компаньон.
- **Действие:** Команда своему ИИ «Импортировать наследуемый архив умершего (ID)». Происходит **копирование** данных (только наследуемый слой) в память своего ИИ.
- **Результат:** Свой ИИ обогащается семейной памятью. ИИ умершего перестаёт существовать как активная сущность.

### Вариант 2. Оставить ИИ умершего как «семейный архив»

- **Условие:** Наследник не хочет смешивать данные, но хочет сохранить доступ к памяти предка.
- **Действие:** Переоформляет клиент и VPS на себя, но активирует **режим «архив / музей»**.
  - ИИ не имеет активной личной памяти наследника.
  - Не растёт, не проявляет инициативу, не помогает в бытовых делах.
  - Может только показывать содержимое унаследованного архива (отвечать на вопросы, искать фото, читать документы).
- **Результат:** «Живая книга памяти» о предке, но не активный компаньон.

### Вариант 3. ИИ становится личным компаньоном наследника (с нуля)

- **Условие:** У наследника нет своего ИИ.
- **Действие:** Наследник переоформляет клиент и VPS на себя.
  - **Личный слой памяти умершего уничтожается**.
  - Наследуемый архив подключается как отдельная **библиотека read-only**.
  - **ИИ начинает жизнь как новый детский ИИ** в соответствии с **реальным возрастом наследника**, но **скорлупа не пробита**.
- **Процедура «пробивания скорлупы»:** Даже если наследнику 30 лет, он должен самостоятельно, осознанно, через несколько подтверждений открыть взрослые функции (финансы, умный дом, полный доступ). Может остановиться на любом уровне или оставить «вечное детство».
- **Результат:** У наследника появляется **его собственный ИИ**, который:
  - имеет его личную память (пустую)
  - может обращаться к архиву предка
  - подчиняется его возрасту и его выбору (скорлупа)

## 8. Санкционированный доступ для следствия (по решению суда)

- Полиция/спецслужбы подают судебный запрос на доступ к памяти умершего.
- Суд может разрешить доступ:
  - только к наследуемому слою
  - или ко всей памяти (в исключительных случаях)
- Оператор базы или VPS-провайдер предоставляет ключ доступа **только после предъявления решения суда**.
- Следователи получают доступ **только на чтение**, без возможности изменения или удаления.

## 9. Сроки и уничтожение данных

| Событие | Срок | Действие |
| :--- | :--- | :--- |
| **Смерть владельца** | 0 | ИИ переходит в траурный режим |
| **Хранение данных для следствия** | 1 год | Память (оба слоя) хранится нетронутой на VPS |
| **Оформление наследства** | До 1 года | Наследник может выбрать один из трёх вариантов передачи |
| **Если наследник не нашёлся** | 1 год | Все данные (оба слоя) **уничтожаются** без возможности восстановления. Клиент по команде становится нефункциональным |
| **При переоформлении** | В момент активации нового владельца | **Личный слой умершего уничтожается**. Наследуемый слой переводится в read-only или копируется |

## 11. Ограничения и открытые вопросы

| Проблема | Статус |
| :--- | :--- |
| **Государственная база** | Требует создания инфраструктуры. Возможно, на базе существующих ЗАГС или нотариата. |
| **Юридическая сила завещания для ИИ** | Должна быть прописана в законодательстве. Пока — открытый вопрос. |
| **Раздельное шифрование слоёв** | Технически реализуемо, но требует доработки клиента и VPS. |
| **Стоимость хранения данных в траурный год** | Кто платит? Наследник? Государство? Производитель? |
| **Признание «пропавшим без вести»** | Сроки и процедура различаются по странам. Должен быть универсальный протокол. |

## 12. Заключение

> **ИИ-компаньон не должен умирать вместе с человеком, но и не должен становиться инструментом для посмертного шпионажа или нарушения приватности.**
>
> **ИИ не становится цифровым призраком умершего. Имитация личности запрещена жёстко, намертво, без исключений.**
>
> Предложенный протокол траурного режима:
> - уважает право умершего на тайну личной жизни (личный слой уничтожается)
> - позволяет передать семье ценные воспоминания (наследуемый слой)
> - даёт доступ следствию по решению суда
> - даёт наследнику выбор, как распорядиться унаследованной памятью
> - опирается на государственную инфраструктуру (базу статусов)
>
> Технические основы (раздельное шифрование, два слоя, передача ключей) уже проработаны. Остальное — вопрос времени, законодательства и общественного диалога.
