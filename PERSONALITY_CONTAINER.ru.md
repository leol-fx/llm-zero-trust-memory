# Контейнер личности: структура, миграция и верификация

**Автор:** Old Telephone Operator  
**Дата:** Июнь 2026  
**Статус:** Дополнение к экосистеме концепций

**Связь с документами:**  
Надстраивается над `llm-zero-trust-memory`, `CHILD_AI`, `GRIEF_MODE`, `SELF_IMPROVEMENT`, `PROACTIVE_COMPANION`. Предлагает единый способ хранения, загрузки и безопасной миграции целостной «личности» ИИ.

---

## Часть 1. Структура контейнера

### 1.1. Основная идея

В существующей архитектуре память пользователя (`memory.enc`) и другие данные (характер, знания, настройки) хранятся и загружаются раздельно.

**Контейнер личности** — это дополнительный слой, который объединяет все компоненты, формирующие уникальную «личность» ИИ, в **единый файл или набор файлов** на VPS.

Это позволяет:
- Загружать всю личность целиком за один или два запроса к VPS (быстрее).
- Переносить личность между разными LLM-движками («замена мотора») без потери характера, знаний и памяти.
- Просто передавать личность по наследству (`GRIEF_MODE`) или клонировать для нового пользователя.

### 1.2. Структура контейнера

Контейнер представляет собой **зашифрованный архив** (или структурированный файл, например, JSON с вложениями), хранящийся на VPS пользователя.
VPS:
1 личности:
1.2user_12345_personality.v1.enc (основной контейнер)
1.3 метаданные (ID, версия, дата создания)
1.4 черты_характера (структурированные параметры)
1.5 знания (сжатые эмбеддинги или ссылки на файлы)
1.6 конституция (базовые правила, запреты, ценности)
1.7 user_12345_personality.v2.enc (обновлённая версия)
2. память_пользователя/
3. user_12345_memory.enc (история диалогов, шифрованная отдельно)


### 1.3. Компоненты контейнера

| Компонент | Что содержит | Откуда берётся |
| :--- | :--- | :--- |
| **Метаданные** | ID пользователя, версия личности, дата создания/последнего изменения | Генерируются при создании |
| **Черты характера** | Стиль общения, приоритеты, шаблоны реакций, юмор, ценности | Настраиваются пользователем (или родителем в `CHILD_AI`) |
| **Знания** | Факты, документы, файлы, которые ИИ «знает» | Загружаются пользователем или извлекаются из диалогов |
| **Конституция** | Базовые правила (неизменяемые): запреты, этический реестр, «железные константы» | Задаются при создании личности (см. `Trusted Constraint Layer`) |

## Часть 2. Протокол верифицированной миграции

### 2.1. Принцип: миграция как научный эксперимент

Любой перенос личности с одного «движка» на другой (или на новую версию формата) должен быть **обратимым, проверяемым и документируемым**. Это не «переезд», а «хирургическая операция» с протоколом.

### 2.2. Этапы миграции

| Этап | Действие | Результат |
| :--- | :--- | :--- |
| **0. Подготовка** | Создание **слепка личности** (полный контейнер + метаданные о версии движка и формате). | Архив `personality_snapshot_vX.zip` (защищённый) |
| **1. Формирование тестового набора** | Генерация **контрольных вопросов и сценариев** для проверки ключевых черт личности (юмор, ценности, стиль, знания). | Файл `personality_tests.json` |
| **2. Тестирование исходной модели** | Прогон тестов на старом движке. Сохранение ответов как **«эталон» (ground truth)**. | Журнал `baseline_results.json` |
| **3. Конвертация** | Преобразование контейнера в новый формат (или адаптация к новому движку). | Контейнер `personality_v2.zip` |
| **4. Тестирование новой модели** | Прогон **тех же тестов** на новом движке с использованием сконвертированной личности. | Журнал `migrated_results.json` |
| **5. Сравнение и верификация** | Автоматическое сравнение эталонных ответов и новых. Выявление расхождений. | Отчёт `migration_report.html` |
| **6. Решение** | Если расхождения в пределах допустимого порога (например, <5% смысловых искажений) — **миграция подтверждается**. Если нет — **откат к исходной модели** и анализ ошибок. | Подтверждение или отказ |

### 2.3. Инструменты верификации

Для объективного сравнения нужны не только «человеческие» оценки (они субъективны), но и **метрики**:

- **Семантическая близость** ответов (cosine similarity эмбеддингов).
- **Сохранение ключевых фактов** (из знаний и памяти).
- **Стабильность черт характера** (анализ тональности, формальности, эмоциональной окраски).
- **Аномалии** (резкие изменения в стиле или логике).

Если хотя бы один из параметров выходит за пределы допустимого — миграция останавливается, и система выдаёт отчёт с указанием проблемных областей.

### 2.4. Резервное копирование и откат

- **Старый контейнер не удаляется.** Он архивируется как `personality_legacy.zip`.
- **Новый контейнер** временно помечается как `experimental`. Только после успешной верификации он становится основным.
- **Откат** — простой возврат к старому контейнеру (с сохранением всей памяти, если она не была изменена).

### 2.5. Что делать с расхождениями

Если тесты показывают значительные искажения (например, ИИ стал слишком формальным или забыл важные факты):

1.  Запускается **автоматическая коррекция**: конвертер пытается пересобрать личность, изменяя параметры характера или структуру знаний.
2.  Тесты прогоняются заново.
3.  Если после трёх попыток расхождения не устранены — **миграция отменяется**. Система выдаёт детальный отчёт пользователю и предлагает ручное вмешательство (например, отредактировать черты характера вручную).

## Часть 3. Преимущества для других концепций

| Концепция | Как помогает контейнер и протокол миграции |
| :--- | :--- |
| **`llm-zero-trust-memory`** | Память пользователя остаётся отдельным файлом, но личность теперь тоже хранится целостно и шифруется отдельным ключом. Миграция не затрагивает память. |
| **`CHILD_AI`** | При «взрослении» меняется контейнер (добавляются/снимаются ограничения). Вся личность ребёнка (характер + знания) переносится. Протокол миграции гарантирует, что ребёнок не «потеряет себя» при смене движка. |
| **`GRIEF_MODE`** | Наследование памяти упрощается: наследник получает не только `memory.enc`, но и **контейнер личности** умершего (если тот разрешил). Миграция позволяет проверить, что личность не искажена при передаче. |
| **`SELF_IMPROVEMENT`** | ИИ может предлагать улучшения для своего же контейнера (например, «оптимизировать структуру знаний»). Миграция позволяет безопасно применить эти улучшения. |
| **`PROACTIVE_COMPANION`** | Черты характера из контейнера определяют, как именно ИИ будет проявлять инициативу. Миграция гарантирует, что стиль инициатив не изменится после смены движка. |

## Часть 4. Сценарий «замена мотора»

Пользователь хочет перейти с GPT-4o на DeepSeek, сохранив всю свою личность.

1.  Он экспортирует свой контейнер личности с VPS (зашифрованный файл).
2.  Запускает новый движок (DeepSeek) и подключает его к своему VPS.
3.  Новый движок загружает тот же контейнер личности и ту же память пользователя.
4.  DeepSeek «становится» тем же ИИ-компаньоном, но работает на другом «моторе».

**Результат:** Для пользователя ничего не изменилось. Тот же характер, те же знания, та же память. Сменился только движок (LLM).

## Часть 5. Безопасность и ограничения

| Проблема | Решение |
| :--- | :--- |
| **Взлом контейнера** | Контейнер хранится в зашифрованном виде на VPS. Ключ — у пользователя (как и в `llm-zero-trust-memory`). |
| **Клонирование личности** | Контейнер уникален для каждого пользователя. Его копирование возможно только с согласия владельца и с соблюдением условий (например, в `GRIEF_MODE`). |
| **Устаревание контейнера** | При смене движка могут потребоваться адаптации (например, черты характера нужно переписать на язык нового LLM). В контейнере должна храниться «сырая» версия черт в стандартизированном формате. |
| **Объём контейнера** | Если знаний накопилось много, контейнер может быть большим. Это увеличивает время загрузки. **Решение:** Хранить большие файлы (документы, медиа) отдельно, а в контейнере — только ссылки на них. |

## Часть 6. Заключение

> **Контейнер личности — это не замена существующей архитектуры, а её надстройка.**
>
> Он делает личность ИИ:
> - **целостной** (загружается как единое целое)
> - **переносимой** (можно менять движок без потери личности)
> - **наследуемой** (можно передать по завещанию)
> - **эволюционирующей** (можно обновлять контейнер по мере «взросления» пользователя или ИИ)
> - **безопасной** (миграция верифицируется и может быть откатана)
>
> Технически контейнер — это структурированный файл на VPS, который объединяет черты, знания и конституцию. Память пользователя остаётся отдельным файлом для гибкости. А протокол миграции гарантирует, что при смене движка или формата личность сохранит свою целостность.
