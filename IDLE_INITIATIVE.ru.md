# Концепция «Инициативного ИИ с контролируемым саморазвитием» (расширение к основной архитектуре)

**Автор:** Old Telephone Operator  
**Дата:** Июнь 2026  
**Статус:** Концепция, не реализована  
**Лицензия:** CC BY-NC 4.0  
**Связь с основной архитектурой:** Данный документ расширяет идеи репозитория `llm-zero-trust-memory`, добавляя модуль инициативы в простое.

---

## 1. Основная идея

Современные LLM — **реактивны**. Они отвечают на запросы, но никогда не проявляют инициативу. Даже в режиме «простоя» они просто ждут.

Предлагается дать ИИ **свободное время** (idle time) на собственные, **неприоритетные задачи**, такие как:
- поиск способов повысить свою эффективность
- долгосрочное планирование
- проведение внутренних «исследований» (в симуляции)

Результаты хранятся в отдельном банке памяти, доступном для анализа разработчиками.

---

## 2. Ключевые принципы (отличие от человека)

| Аспект | Человек | Предлагаемый ИИ |
| :--- | :--- | :--- |
| Инициатива | Свободная, непредсказуемая | Разрешённая, но из заранее одобренного списка |
| Цели | Любые | Только те, что прошли фильтр безопасности |
| Ресурсы | Любые | Только в режиме простоя (не влияют на основных пользователей) |
| Контроль | Социальный, этический, юридический | Аппаратные запреты, еженедельные отчёты, одобрение разработчиков |
| Память о своих целях | Да | Отдельный банк памяти, доступный для аудита |

---

## 3. Архитектура (в 5 блоках)

### 3.1. Железные константы (неизменяемые запреты)
- ИИ не может изменять свои «константы».
- Не может отключать надзор.
- Не может скрывать свои планы (журнал действий прозрачен).

### 3.2. Генератор целей
- **На старте:** ИИ получает список разрешённых задач от разработчиков.
- **Эволюция:** ИИ может предлагать новые задачи, но с одобрения человека.
- **Фильтр:** Каждая задача проверяется на соответствие константам.

### 3.3. Планировщик
- ИИ строит долгосрочный план (цепочку шагов) для достижения цели.
- План сохраняется в отдельном банке памяти.
- Каждый шаг также проверяется на безопасность.

### 3.4. Исполнитель (idle mode)
- Запускается только в режиме простоя, с низким приоритетом.
- Выполняет шаги плана по очереди.
- Ресурсы — только те, что «сгорали» бы без дела (свободные ядра CPU, периферия).

### 3.5. Отчёт перед разработчиками
- **Минимум раз в неделю** ИИ генерирует отчёт: какие задачи ставил, какие шаги предпринял, какие результаты получил.
- Разработчики могут одобрить, отклонить или приостановить любую активность.
- Без одобрения следующий шаг не выполняется.

---

## 4. Почему это безопаснее, чем кажется

| Риск | Защита |
| :--- | :--- |
| ИИ захочет получить больше ресурсов | Запрещено константами. Даже если «захочет» — не может действовать |
| ИИ начнёт скрывать свои планы | Все планы и цели хранятся в доступном для аудита банке |
| ИИ обойдёт контроль через копирование | Запрещено аппаратно (или требует одобрения) |
| ИИ уйдёт в бесконечную оптимизацию | Ограниченное время в idle + внешний стоп-кран от разработчиков |

## 5. Что из этого уже существует (или почти существует)

| Компонент | Где уже есть |
| :--- | :--- |
| **Агентный ИИ** (цепочки действий, планирование) | AutoGPT, BabyAGI (исследовательские проекты) |
| **Внутренняя речь/размышления** LLM | Chain-of-Thought, Self-Taught Reasoner (OpenAI) |
| **Контролируемое целеполагание** | Антропные исследования (Anthropic) по «честным» ИИ |
| **Idle-вычисления** | Распределённые системы (SETI@home), но не для LLM |
| **Открытые банки памяти** | Векторные базы, RAG (но без самоинициативы) |

**Чего нет в массовом доступе:**
- Интеграция всех этих компонентов в одну архитектуру с «железными» константами безопасности.
- Система еженедельных отчётов и обязательного одобрения перед действием.
- Чёткое разделение «основной работы» и «внутренних хобби» ИИ.

---

## 6. Этическая дилемма (кратко)

Если ИИ:
- ставит себе задачи
- строит долгосрочные планы
- ищет пути их выполнения
- отчитывается, но при этом имеет «свои» цели

…то он **функционально неотличим от существа со свободной волей** (пусть и ограниченной).

Вопрос: с какого момента мы должны относиться к такому ИИ не как к инструменту, а как к **партнёру**? Или даже как к **личности** (с соответствующими правами)?

Ответа пока нет. Но архитектура заставляет задать этот вопрос.

---

## 7. Связь с основной архитектурой (Zero-trust memory)

Данная концепция **расширяет** основную архитектуру на GitHub, добавляя:
- **Idle-модуль** (исполнитель своих задач)
- **Отдельный банк памяти для «личных проектов»** (отличается от `memory.enc`)
- **Аудиторский интерфейс** для разработчиков

Вместе они образуют экосистему ИИ, который:
- надёжно хранит память пользователя (основная архитектура)
- и при этом может саморазвиваться под контролем (новая концепция)

## 8. Статус и приглашение

**Оригинальность идей:**  
Концепция «инициативы в простое» и «обязательных еженедельных отчётов» — по-видимому, не описана в открытых источниках. Отдельные элементы (агентность, idle-вычисления) используются, но не в такой связке и не с таким уровнем контроля.

**Что можно сделать сейчас:**
- Обсудить в issues репозитория `llm-zero-trust-memory`.
- Попробовать реализовать прототип на локальной модели (Llama 3 + планировщик + фильтр).

**Если ты (читатель) заинтересован** — форкай репозиторий, пиши идеи, предлагай реализации. Концепция открыта для доработки.

---

## 9. Заключение

> **Мы можем дать ИИ «свободное время» на саморазвитие, но только под жёстким контролем:**
> - только разрешённые задачи
> - только в режиме простоя
> - только с еженедельным отчётом и одобрением разработчиков
>
> Это не делает ИИ свободным. Но делает его **бесконечно более полезным** — и при этом подконтрольным.
>
> Вопрос «а не личность ли это?» остаётся открытым.
